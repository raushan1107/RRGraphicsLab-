Demo 1 · Module 1 Concept Visual
The Rendering Pipeline Walkthrough
Before writing a single line of code, understand what the GPU does at every stage. This demo is a deep visual walkthrough of the 7-stage rendering pipeline — the assembly line that converts your float arrays into coloured pixels on screen.
🎓 Theory Demo ⏱ 15 min 📋 No code required
💡 The Core Idea

The GPU is a pipeline — like a car assembly line. Raw material (vertex positions) enters one end, finished product (coloured pixels) exits the other. Each station does exactly one job. You only write code for two stations — the Vertex Shader and Fragment Shader. The rest is automatic GPU hardware.

The 7-Stage Rendering Pipeline

What Each Stage Does — In Plain English

VERTEX SHADER
Runs once per vertex on the GPU
Takes a 3D world position and converts it to a 2D screen position. This is where camera and projection math happens. If you have 3 vertices (triangle), this runs 3 times simultaneously on the GPU.
RASTERIZATION
Automatic: Converts triangles → pixel candidates
GPU determines which pixels on screen a triangle covers. A triangle covering 10,000 pixels produces 10,000 "fragments". This is why the GPU needs thousands of cores — it processes all fragments in parallel.
FRAGMENT SHADER
Runs once per pixel candidate on the GPU
You decide the final colour of each pixel. This is where texture lookups, lighting calculations, shadows, and post-processing happen. For a 1080p screen at 60fps covering the whole display, this runs 124 million times per second.
✅ Naval Use Case — Radar Display

A ship radar shows 200 contacts. Each contact icon is a small triangle. Vertex shader: positions each contact at correct bearing/range on screen. Rasterizer: determines which pixels each contact covers. Fragment shader: colours green for friendly, red for hostile based on a classification uniform passed from C++. At 60fps the whole pipeline runs 60 times per second while contacts update from radar hardware.

Step 2 — CMakeLists.txt (Build System Config)

💡 What is CMakeLists.txt?

CMake is a build system generator. Instead of manually configuring Visual Studio project settings (which would require 30 clicks per project), you write one text file. CMake reads it and generates the VS project files automatically. Think of it as a recipe: "here are my source files, here are my libraries, here's how to link them."

CMakeLists.txt
CMake 
# Minimum CMake version — older versions lack features we need
cmake_minimum_required(VERSION 3.20)

# Project name — this becomes your .sln and .vcxproj name in VS
project(M02_Window)

# Tell CMake to compile all .cpp files as C++17
# C++17 gives us structured bindings, std::filesystem, and more
set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# Find OpenGL — it's already on your system (GPU driver installs it)
# REQUIRED means CMake will fail with a clear error if not found
find_package(OpenGL REQUIRED)

# Read our environment variables to find library locations
# $ENV{} reads Windows environment variables you set earlier
set(GLFW_DIR $ENV{GLFW_DIR})    # should be C:\libs\glfw
set(GLEW_DIR $ENV{GLEW_DIR})    # should be C:\libs\glew
set(GLM_DIR  $ENV{GLM_DIR})     # should be C:\libs\glm

# Tell compiler where to find header files (.h/.hpp)
# include_directories adds these paths to the compiler's search list
include_directories(
    ${GLFW_DIR}/include          # GLFW/glfw3.h
    ${GLEW_DIR}/include          # GL/glew.h
    ${GLM_DIR}                   # glm/glm.hpp
)

# Define our executable target — "M02_Window.exe" built from main.cpp
add_executable(M02_Window src/main.cpp)

# Link the libraries — attach .lib files to our executable
# Without this, you get "unresolved external symbol" linker errors
target_link_libraries(M02_Window
    OpenGL::GL                                          # opengl32.lib from Windows SDK
    ${GLFW_DIR}/lib-vc2022/glfw3.lib                   # GLFW (pre-compiled for VS2022)
    ${GLEW_DIR}/lib/Release/x64/glew32s.lib            # GLEW static library (the 's' = static)
)

Step 3 — main.cpp (Complete Window Program)

💡 What GLFW Creates for You

GLFW creates three things: (1) A native OS window — the same window.h/Win32 window as any other app. (2) An OpenGL context — the stateful workspace where all GL operations execute. (3) An input event queue — collects keyboard/mouse events between frames. Without a context, every gl* call crashes silently.

src/main.cpp
C++ 
// GLEW must ALWAYS be included before GLFW — it overrides some GL headers
// GLEW_STATIC = use the static (.lib) version, no .dll needed at runtime
#define GLEW_STATIC
#include <GL/glew.h>

#include <GLFW/glfw3.h>   // Window creation, context, input
#include <iostream>          // std::cout for printing GPU info

// ─────────────────────────────────────────────────────────────────
// CALLBACK: called automatically by GLFW when window is resized
// GLFW passes: the window, new width, new height
// We tell OpenGL the new drawable area (viewport = what GL draws into)
// ─────────────────────────────────────────────────────────────────
void framebuffer_size_callback(GLFWwindow* window, int width, int height) {
    // glViewport maps GL's NDC space (-1 to 1) to this pixel rectangle
    // x=0, y=0 = bottom-left of window, width × height = full window
    glViewport(0, 0, width, height);
}

// ─────────────────────────────────────────────────────────────────
// FUNCTION: process keyboard input every frame
// Called inside the render loop so it's checked 60 times/second
// ─────────────────────────────────────────────────────────────────
void processInput(GLFWwindow* window) {
    // GLFW_KEY_ESCAPE = the Escape key. GLFW_PRESS = key is currently held
    // glfwSetWindowShouldClose sets the "please quit" flag
    // The render loop checks this flag and exits when it's true
    if (glfwGetKey(window, GLFW_KEY_ESCAPE) == GLFW_PRESS)
        glfwSetWindowShouldClose(window, true);
}

int main() {
    // ── STEP 1: Initialise GLFW ──────────────────────────────────────
    // Must be called before any other GLFW function
    // Returns GLFW_TRUE if successful, GLFW_FALSE if failed
    if (!glfwInit()) {
        std::cerr << "Failed to initialise GLFW\n";
        return -1;
    }

    // ── STEP 2: Configure OpenGL Context Hints ───────────────────────
    // These hints tell GLFW what kind of context to create
    // Must be set BEFORE glfwCreateWindow

    // Request OpenGL version 3.3 — our minimum for modern features
    glfwWindowHint(GLFW_CONTEXT_VERSION_MAJOR, 3);
    glfwWindowHint(GLFW_CONTEXT_VERSION_MINOR, 3);

    // Core profile = modern OpenGL only (no deprecated functions like glBegin)
    // Compat profile = all old functions still available (legacy mode)
    glfwWindowHint(GLFW_OPENGL_PROFILE, GLFW_OPENGL_CORE_PROFILE);

    // ── STEP 3: Create the Window + Context ─────────────────────────
    // Parameters: width, height, title, monitor (NULL=windowed), share-context
    // Returns a GLFWwindow* handle — everything goes through this
    GLFWwindow* window = glfwCreateWindow(800, 600, "RR Graphics Lab - Module 2", NULL, NULL);
    if (!window) {
        std::cerr << "Failed to create GLFW window\n";
        glfwTerminate();  // Clean up GLFW before exiting
        return -1;
    }

    // ── STEP 4: Make Context Current ────────────────────────────────
    // "Current" means: all gl* calls on this thread now go to this context
    // CRITICAL: GLEW init will fail if no context is current
    glfwMakeContextCurrent(window);

    // ── STEP 5: Register Resize Callback ────────────────────────────
    // Whenever the window is resized, GLFW calls framebuffer_size_callback
    // This keeps the GL viewport matching the actual window size
    glfwSetFramebufferSizeCallback(window, framebuffer_size_callback);

    // ── STEP 6: Initialise GLEW ──────────────────────────────────────
    // GLEW finds all OpenGL function pointers in the GPU driver
    // MUST come after glfwMakeContextCurrent — needs an active context
    // glewExperimental = true enables newer extension features
    glewExperimental = GL_TRUE;
    if (glewInit() != GLEW_OK) {
        std::cerr << "Failed to initialise GLEW\n";
        return -1;
    }

    // Print GPU info so students can see their hardware
    std::cout << "GPU: "            << glGetString(GL_RENDERER) << "\n";
    std::cout << "OpenGL Version: " << glGetString(GL_VERSION)  << "\n";

    // ── STEP 7: THE RENDER LOOP ──────────────────────────────────────
    // Runs continuously until the window is closed
    // glfwWindowShouldClose returns true when ESC is pressed or X clicked
    while (!glfwWindowShouldClose(window)) {

        // ① PROCESS INPUT — check keyboard/mouse events
        processInput(window);

        // ② CLEAR — erase the previous frame
        // GL_COLOR_BUFFER_BIT clears the colour buffer (the image)
        // Use GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT when 3D is added
        glClearColor(0.05f, 0.10f, 0.20f, 1.0f); // R, G, B, Alpha (navy blue)
        glClear(GL_COLOR_BUFFER_BIT);

        // ③ DRAW — we'll add drawing calls here from Demo 3 onwards
        // (nothing to draw yet — just a coloured window)

        // ④ SWAP BUFFERS — show the completed frame
        // OpenGL uses double buffering: you draw to the BACK buffer
        // then swap to make it the FRONT buffer (visible on screen)
        // Without this call: screen stays black even though you drew
        glfwSwapBuffers(window);

        // ⑤ POLL EVENTS — process queued input events
        // Without this: window appears frozen, can't respond to clicks
        // Alternative: glfwWaitEvents() — sleeps until an event arrives
        glfwPollEvents();
    }

    // ── CLEANUP ─────────────────────────────────────────────────────
    // Free all GLFW resources (windows, contexts, input devices)
    glfwTerminate();
    return 0;
}

Demo 3 · Module 3GPU Memory
VBO — Uploading Vertex Data to the GPU
The VBO (Vertex Buffer Object) is how data travels from your C++ array in RAM to GPU VRAM. Without this, the GPU has no geometry to draw. Three function calls — every OpenGL application uses this exact pattern.
💻 Add to Demo 2⏱ 20 min
CPU RAM → GPU VRAM — The VBO Transfer
  Your C++ code (CPU RAM)               GPU VRAM (after glBufferData)
  ┌────────────────────────┐            ┌────────────────────────────┐
  │ float vertices[] = {   │            │  VBO #1                    │
  │  -0.5f, -0.5f, 0.0f,  │ ─────────→ │  bytes: [BF000000 BF000000 │
  │   0.5f, -0.5f, 0.0f,  │  PCIe Bus  │   00000000 3F000000 ...]   │
  │   0.0f,  0.5f, 0.0f   │  (fast!)   │  (same data, GPU-readable) │
  │ };                     │            │                            │
  └────────────────────────┘            └────────────────────────────┘
  This array can be deleted             GPU reads this every draw call
  after glBufferData() — GPU            without going back to CPU RAM
  has its own copy now                  (much faster!)
VBO Creation Pattern — add to main.cpp before the render loop
C++ 
// ─── VERTEX DATA ────────────────────────────────────────────────
// Define 3 vertices of a triangle in NDC (Normalized Device Coords)
// NDC: x and y range from -1.0 to +1.0 on screen
// Each vertex = 3 floats: X position, Y position, Z position
// Z=0.0 means on the screen plane (we'll add depth in Module 4)
float vertices[] = {
    // X       Y       Z
    -0.5f, -0.5f,  0.0f,   // bottom-left  vertex
     0.5f, -0.5f,  0.0f,   // bottom-right vertex
     0.0f,  0.5f,  0.0f    // top-center   vertex
};

// ─── VBO CREATION ───────────────────────────────────────────────
// Step 1: Generate (create) one buffer object on the GPU
// OpenGL gives us an ID number (like a key/handle) stored in VBO
// We use this ID for all future operations on this buffer
unsigned int VBO;
glGenBuffers(1, &VBO);
// glGenBuffers(n, array)
//   n     = how many buffers to create (we need 1)
//   array = where to store the generated IDs
// After this: VBO = some integer like 1, 2, 3... (GPU-assigned ID)

// Step 2: Bind — select this VBO as the active array buffer
// OpenGL is a state machine: you "select" a thing, then operate on it
// GL_ARRAY_BUFFER = target slot for vertex data
// All subsequent buffer operations work on this bound buffer
glBindBuffer(GL_ARRAY_BUFFER, VBO);
// glBindBuffer(target, id)
//   GL_ARRAY_BUFFER = vertex attribute data (positions, colours, UVs)
//   VBO             = the ID we just generated
// You can CHANGE buffer later by binding a different ID to same target

// Step 3: Upload our vertex data from CPU RAM to GPU VRAM
// After this call, the GPU has its own copy and you don't need vertices[]
glBufferData(GL_ARRAY_BUFFER, sizeof(vertices), vertices, GL_STATIC_DRAW);
// glBufferData(target, size, data, usage)
//   GL_ARRAY_BUFFER   = same target as above (operates on bound buffer)
//   sizeof(vertices)  = total bytes: 9 floats × 4 bytes = 36 bytes
//   vertices          = pointer to the start of our float array
//   GL_STATIC_DRAW    = hint: data won't change (lets GPU optimise placement)
//   Other hints: GL_DYNAMIC_DRAW (changes often), GL_STREAM_DRAW (changes every frame)
⚠️ Important — Names You Control vs Names That Are Fixed

The variable name VBO is something YOU choose — you could name it myBuffer, triangleVBO, or buf1. What's fixed is the function name (glGenBuffers), the target (GL_ARRAY_BUFFER), and the usage hint (GL_STATIC_DRAW). These come from the OpenGL specification and are the same in every C++ OpenGL program on every machine.

Demo 4 · Module 3Data Layout
VAO + glVertexAttribPointer Decoded
The VAO records how to interpret raw VBO bytes. The glVertexAttribPointer call is the most confusing in OpenGL — we decode every single parameter with a diagram.
💻 Add to Demo 3⏱ 25 min
glVertexAttribPointer — Every Parameter Explained
  Memory layout of our vertices[] array (position only — 3 floats per vertex):

  Byte offset:  0    4    8   12   16   20   24   28   32
                │    │    │    │    │    │    │    │    │
  Raw memory:  [X0] [Y0] [Z0] [X1] [Y1] [Z1] [X2] [Y2] [Z2]
                ├──────────┤   ├──────────┤   ├──────────┤
                  Vertex 0        Vertex 1        Vertex 2
                  12 bytes        12 bytes        12 bytes

  Stride = 12 bytes (distance from start of one vertex to start of next)
  Offset = 0 bytes  (position starts at byte 0 of each vertex)

  glVertexAttribPointer(0,    3,        GL_FLOAT, GL_FALSE, 12,   (void*)0);
                        │     │         │         │         │     │
                        │     │         │         │         │     └─ Byte offset: where THIS attribute
                        │     │         │         │         │          starts inside one vertex (0 = start)
                        │     │         │         │         └─ Stride: gap between this attr in consecutive
                        │     │         │         │              vertices (12 bytes = 3 floats × 4 bytes)
                        │     │         │         └─ Normalized: GL_FALSE = use raw values (not 0-1 range)
                        │     │         └─ Type: each component is a 32-bit float (4 bytes)
                        │     └─ Count: 3 components per vertex (x, y, z)
                        └─ Attribute index 0: matches "layout(location=0)" in vertex shader

  glEnableVertexAttribArray(0) — enables attribute 0 so GPU reads it
VAO setup — add BEFORE VBO creation in main.cpp
C++ 
// ─── CREATE VAO FIRST (before VBO) ──────────────────────────────
// VAO = Vertex Array Object. It records ALL the state needed to
// describe how to read vertices. Bind it, then set up VBO + attribs.
// When you bind it again later, all that state is restored instantly.
unsigned int VAO;
glGenVertexArrays(1, &VAO);
glBindVertexArray(VAO);   // RECORDING STARTS: everything below is remembered

// ... (VBO creation and glBufferData go here as in Demo 3) ...

// ─── TELL GPU HOW TO INTERPRET THE VBO BYTES ────────────────────
// This is the call every beginner struggles with — let's decode it
glVertexAttribPointer(
    0,            // Attribute index 0 — must match "layout(location=0)" in vertex shader
    3,            // 3 components per vertex attribute (x, y, z)
    GL_FLOAT,     // Each component is a 32-bit floating point number
    GL_FALSE,     // Don't normalize — use raw float values as-is
                  // GL_TRUE would map integers to 0.0-1.0 range
    3 * sizeof(float), // Stride: 12 bytes between consecutive positions
                       // = gap from vertex[0].x to vertex[1].x in memory
    (void*)0       // Offset: 0 bytes — position starts at byte 0 of each vertex
                  // Cast to void* because this can be a byte offset (legacy API design)
);

// Enable the attribute so the vertex shader actually receives it
// Disabled by default — you must enable each attribute you want to use
glEnableVertexAttribArray(0);  // Enable attribute at index 0

// RECORDING STOPS — unbind VAO (good practice, prevents accidental modification)
glBindVertexArray(0);

// ─── IN THE RENDER LOOP, drawing is now this simple: ─────────────
// glBindVertexArray(VAO);        ← restore all state in one call
// glDrawArrays(GL_TRIANGLES, 0, 3); ← draw 3 vertices as a triangle

💡 Shaders — Tiny Programs Running on the GPU

A shader is a program written in GLSL (OpenGL Shading Language). It looks like C but runs on the GPU instead of the CPU. You write the shader source code as a string in your C++ code, send it to OpenGL which compiles it on the GPU driver, then link shaders into a shader program that runs on every vertex or every pixel during drawing.

src/main.cpp — Complete Triangle with Shaders
C++ 
#define GLEW_STATIC
#include <GL/glew.h>
#include <GLFW/glfw3.h>
#include <iostream>

// ─── VERTEX SHADER SOURCE (GLSL) ────────────────────────────────
// This is a raw string literal (C++11). R"(...)" preserves newlines.
// The GPU compiles this at runtime — it runs once PER VERTEX.
//
// #version 330 core → use GLSL 3.30, core profile (modern OpenGL)
// layout(location=0) → attribute index must match glVertexAttribPointer(0,...)
// in vec3 aPos → "in" = input from VBO. vec3 = 3 floats. "aPos" = your name choice.
// gl_Position → BUILT-IN output. Must be set. GPU uses it for rasterization.
// vec4(aPos, 1.0) → promote vec3 to vec4 (homogeneous coords, w=1.0 for positions)
const char* vertexShaderSource = R"(
    #version 330 core
    layout(location = 0) in vec3 aPos;

    void main() {
        gl_Position = vec4(aPos.x, aPos.y, aPos.z, 1.0);
    }
)";

// ─── FRAGMENT SHADER SOURCE (GLSL) ──────────────────────────────
// Runs once PER FRAGMENT (pixel candidate) — after rasterization.
// out vec4 FragColor → your output variable. Name is up to you.
//   But it must be "out" and vec4 (RGBA). 
// vec4(1.0, 0.5, 0.2, 1.0) → orange: R=1(full), G=0.5(half), B=0.2(low), A=1(opaque)
const char* fragmentShaderSource = R"(
    #version 330 core
    out vec4 FragColor;

    void main() {
        FragColor = vec4(1.0f, 0.5f, 0.2f, 1.0f);
    }
)";

void framebuffer_size_callback(GLFWwindow* w, int width, int height) { glViewport(0,0,width,height); }
void processInput(GLFWwindow* w) { if(glfwGetKey(w,GLFW_KEY_ESCAPE)==GLFW_PRESS) glfwSetWindowShouldClose(w,true); }

int main() {
    // ── Window setup (same as Demo 2) ────────────────────────────
    glfwInit();
    glfwWindowHint(GLFW_CONTEXT_VERSION_MAJOR, 3);
    glfwWindowHint(GLFW_CONTEXT_VERSION_MINOR, 3);
    glfwWindowHint(GLFW_OPENGL_PROFILE, GLFW_OPENGL_CORE_PROFILE);
    GLFWwindow* window = glfwCreateWindow(800, 600, "First Triangle", NULL, NULL);
    glfwMakeContextCurrent(window);
    glfwSetFramebufferSizeCallback(window, framebuffer_size_callback);
    glewExperimental = GL_TRUE;
    glewInit();

    // ── COMPILE VERTEX SHADER ────────────────────────────────────
    unsigned int vertexShader = glCreateShader(GL_VERTEX_SHADER);
    // glCreateShader(type) → creates a shader object, returns its ID
    // GL_VERTEX_SHADER = runs once per vertex
    // GL_FRAGMENT_SHADER = runs once per pixel candidate

    glShaderSource(vertexShader, 1, &vertexShaderSource, NULL);
    // glShaderSource(shader, count, source_array, lengths_array)
    //   shader       = our shader object ID
    //   count        = 1 (we have 1 source string)
    //   source_array = pointer to our GLSL source string
    //   lengths      = NULL (GL calculates length from null-terminator)

    glCompileShader(vertexShader);
    // The GPU DRIVER compiles the GLSL source at runtime (not CMake time!)
    // Always check for errors after compilation:
    int success; char infoLog[512];
    glGetShaderiv(vertexShader, GL_COMPILE_STATUS, &success);
    if (!success) {
        glGetShaderInfoLog(vertexShader, 512, NULL, infoLog);
        std::cerr << "Vertex shader error: " << infoLog << "\n";
    }

    // ── COMPILE FRAGMENT SHADER (same pattern) ───────────────────
    unsigned int fragmentShader = glCreateShader(GL_FRAGMENT_SHADER);
    glShaderSource(fragmentShader, 1, &fragmentShaderSource, NULL);
    glCompileShader(fragmentShader);
    glGetShaderiv(fragmentShader, GL_COMPILE_STATUS, &success);
    if (!success) {
        glGetShaderInfoLog(fragmentShader, 512, NULL, infoLog);
        std::cerr << "Fragment shader error: " << infoLog << "\n";
    }

    // ── LINK SHADER PROGRAM ─────────────────────────────────────
    // A shader program = linked vertex + fragment shader pair
    // This is what you activate with glUseProgram() before drawing
    unsigned int shaderProgram = glCreateProgram();
    glAttachShader(shaderProgram, vertexShader);
    glAttachShader(shaderProgram, fragmentShader);
    glLinkProgram(shaderProgram);
    // Individual shaders can be deleted after linking — GPU has the linked program
    glDeleteShader(vertexShader);
    glDeleteShader(fragmentShader);

    // ── VERTEX DATA + VBO + VAO ──────────────────────────────────
    float vertices[] = { -0.5f,-0.5f,0.0f,  0.5f,-0.5f,0.0f,  0.0f,0.5f,0.0f };

    unsigned int VAO, VBO;
    glGenVertexArrays(1, &VAO);
    glGenBuffers(1, &VBO);

    glBindVertexArray(VAO);                         // start recording
      glBindBuffer(GL_ARRAY_BUFFER, VBO);
      glBufferData(GL_ARRAY_BUFFER, sizeof(vertices), vertices, GL_STATIC_DRAW);
      glVertexAttribPointer(0, 3, GL_FLOAT, GL_FALSE, 3*sizeof(float), (void*)0);
      glEnableVertexAttribArray(0);
    glBindVertexArray(0);                           // stop recording

    // ── RENDER LOOP ──────────────────────────────────────────────
    while (!glfwWindowShouldClose(window)) {
        processInput(window);

        glClearColor(0.1f, 0.1f, 0.15f, 1.0f);
        glClear(GL_COLOR_BUFFER_BIT);

        // ① Activate our shader program — all subsequent draws use these shaders
        glUseProgram(shaderProgram);

        // ② Bind VAO — restores VBO binding + attribute layout in one call
        glBindVertexArray(VAO);

        // ③ Draw! 
        // GL_TRIANGLES = group every 3 vertices into a triangle
        // 0 = start at index 0 in the VBO
        // 3 = draw 3 vertices (= 1 triangle)
        glDrawArrays(GL_TRIANGLES, 0, 3);

        glfwSwapBuffers(window);
        glfwPollEvents();
    }

    // ── CLEANUP ──────────────────────────────────────────────────
    glDeleteVertexArrays(1, &VAO);
    glDeleteBuffers(1, &VBO);
    glDeleteProgram(shaderProgram);
    glfwTerminate();
    return 0;
}

  • Break the shader intentionally: Remove the semicolon from a GLSL line. Run → see the compiler error message printed to terminal.
  • Draw with GL_POINTS: Change GL_TRIANGLES → GL_POINTS → see 3 separate dots. Change to GL_LINE_LOOP for an outline.
Demo 6 · Module 3Index Buffers
EBO — Rectangle from 4 Vertices (not 6)
Draw a rectangle efficiently using an Element Buffer Object. Store 4 unique corners + an index list. The GPU assembles 2 triangles from those 4 points, avoiding duplicated vertex data.
💻 Extend Demo 5⏱ 20 min
Rectangle = 2 Triangles — Index Buffer Strategy
  4 Unique vertices (stored in VBO):     Index data (stored in EBO):
  0 (-0.5,  0.5)   1 (0.5,  0.5)       { 0, 1, 3,    ← Triangle 1: top-left, top-right, bottom-left
                                           1, 2, 3 }   ← Triangle 2: top-right, bottom-right, bottom-left
  3 (-0.5, -0.5)   2 (0.5, -0.5)

  Triangle 1: vertices 0, 1, 3 (upper-left triangle)
  Triangle 2: vertices 1, 2, 3 (lower-right triangle)
  Together they form a solid rectangle.

  Without EBO: 6 vertices needed (2 duplicated) = 6 × 12 bytes = 72 bytes
  With EBO:    4 vertices + 6 indices = 48 + 24 = 72 bytes... BUT for complex
               meshes with 100k+ vertices, EBO saves ~30-40% VRAM.
EBO additions to main.cpp — replace vertices + VAO section
C++ 
// ── 4 unique corner positions (rectangle) ─────────────────────
float vertices[] = {
    // X       Y      Z
     0.5f,  0.5f, 0.0f,   // vertex 0: top-right
     0.5f, -0.5f, 0.0f,   // vertex 1: bottom-right
    -0.5f, -0.5f, 0.0f,   // vertex 2: bottom-left
    -0.5f,  0.5f, 0.0f    // vertex 3: top-left
};

// ── Index list — which vertices make each triangle ─────────────
// Two triangles to fill the rectangle:
// Triangle 1: vertices 0, 1, 3  (top-right, bottom-right, top-left)
// Triangle 2: vertices 1, 2, 3  (bottom-right, bottom-left, top-left)
unsigned int indices[] = {
    0, 1, 3,   // first triangle
    1, 2, 3    // second triangle
};

unsigned int VAO, VBO, EBO;
glGenVertexArrays(1, &VAO);
glGenBuffers(1, &VBO);
glGenBuffers(1, &EBO);    // EBO is just another buffer, different target

glBindVertexArray(VAO);

  glBindBuffer(GL_ARRAY_BUFFER, VBO);
  glBufferData(GL_ARRAY_BUFFER, sizeof(vertices), vertices, GL_STATIC_DRAW);

  // ── EBO: GL_ELEMENT_ARRAY_BUFFER is the target for index data ─
  // When a VAO is bound, binding an EBO to GL_ELEMENT_ARRAY_BUFFER
  // makes the VAO remember this EBO — so it's restored on bind
  glBindBuffer(GL_ELEMENT_ARRAY_BUFFER, EBO);
  glBufferData(GL_ELEMENT_ARRAY_BUFFER, sizeof(indices), indices, GL_STATIC_DRAW);

  glVertexAttribPointer(0, 3, GL_FLOAT, GL_FALSE, 3*sizeof(float), (void*)0);
  glEnableVertexAttribArray(0);

glBindVertexArray(0);

// ── IN THE RENDER LOOP — glDrawElements instead of glDrawArrays ─
glBindVertexArray(VAO);
glDrawElements(
    GL_TRIANGLES,    // primitive type: each 3 indices = 1 triangle
    6,               // count: 6 indices total (2 triangles × 3 indices)
    GL_UNSIGNED_INT, // type of index values in the EBO
    0                // offset into the EBO: start from index 0
);

// WIREFRAME MODE — toggle to see both triangles:
// glPolygonMode(GL_FRONT_AND_BACK, GL_LINE);   ← wireframe
// glPolygonMode(GL_FRONT_AND_BACK, GL_FILL);   ← filled (default)
🧪 Experiments
  • Enable wireframe: Add glPolygonMode(GL_FRONT_AND_BACK, GL_LINE); before glDrawElements → see both triangles as outlines.
  • Change winding order: Change index 0,1,3 → 0,3,1 — if backface culling is on, that triangle disappears.
  • Make a pentagon: Add 5 vertices and 9 indices (3 triangles from a centre point).
Demo 7 · Module 3Dynamic Rendering
Uniforms + Animation — Sending Data to Shaders Every Frame
A uniform is how your C++ code sends data to a GLSL shader every frame. Pass the current time → shader uses sin(time) to animate colour. Your first real-time dynamic rendering.
💻 Extend Demo 5⏱ 20 min
💡 What Is a Uniform?

A uniform is a variable that stays the same value for all vertices and all fragments in one draw call. You set it from C++ using glUniform*() and read it in GLSL. Unlike vertex attributes (which differ per vertex), a uniform is broadcast to every shader invocation. Common uniforms: transformation matrix, time, light position, material colour.

Fragment shader + C++ uniform — add time-based colour animation
C++ + GLSL 
// ─── UPDATED FRAGMENT SHADER ─────────────────────────────────────
// uniform float uTime → receives the time value we'll send each frame
// sin() returns -1 to +1, so (sin()+1)/2 maps to 0.0 to 1.0 for colour
const char* fragmentShaderSource = R"(
    #version 330 core
    out vec4 FragColor;

    uniform float uTime;    // Sent from C++ every frame via glUniform1f

    void main() {
        // sin(uTime) oscillates between -1 and +1
        // (sin(uTime) + 1.0) / 2.0 maps to 0.0 - 1.0
        float r = (sin(uTime * 1.0) + 1.0) / 2.0;
        float g = (sin(uTime * 0.5 + 1.0) + 1.0) / 2.0;
        float b = (sin(uTime * 0.3 + 2.0) + 1.0) / 2.0;
        FragColor = vec4(r, g, b, 1.0);
    }
)";

// ─── IN THE RENDER LOOP (add after glUseProgram) ─────────────────
while (!glfwWindowShouldClose(window)) {
    processInput(window);
    glClearColor(0.1f, 0.1f, 0.15f, 1.0f);
    glClear(GL_COLOR_BUFFER_BIT);

    glUseProgram(shaderProgram);

    // ── SEND TIME UNIFORM ────────────────────────────────────────
    // glfwGetTime() returns seconds since glfwInit() — precise, monotonic
    // glGetUniformLocation finds "uTime" by name in the compiled shader
    // Returns -1 if the name doesn't exist (check this in production)
    float timeVal = (float)glfwGetTime();
    int timeLoc = glGetUniformLocation(shaderProgram, "uTime");
    // glUniform1f → sets ONE float uniform
    // Other variants: glUniform2f (vec2), glUniform3f (vec3),
    //                 glUniform4f (vec4), glUniformMatrix4fv (mat4)
    glUniform1f(timeLoc, timeVal);

    glBindVertexArray(VAO);
    glDrawArrays(GL_TRIANGLES, 0, 3);

    glfwSwapBuffers(window);
    glfwPollEvents();
}
✅ Expected Visual Output
A triangle that smoothly cycles through colours over time.
RGB channels cycle at different speeds (different sin frequencies).
The animation runs continuously without any input needed.
🧪 Experiments
  • Speed up: Change uTime * 1.0 → uTime * 5.0 in the shader — colours cycle much faster.
  • Animate position: Add uniform float uOffset; to vertex shader, apply it as gl_Position = vec4(aPos.x + uOffset, aPos.y, 0, 1), send glUniform1f(offsetLoc, sin(timeVal) * 0.5f) — triangle oscillates left-right.
  • Cache the location: Move glGetUniformLocation BEFORE the loop (it's expensive to call every frame). This is a real optimization pattern.
