Demo 15 · Module 10Vulkan Startup Sequence — Terminal Only
VkInstance, Physical Device, Logical Device — The Root of Every Vulkan Programme
No window. No rendering. Pure startup sequence printed to terminal. Creates a VkInstance with validation layers, enumerates every GPU in the system with its VRAM and driver version, selects the best GPU, finds a graphics queue family, and creates a VkDevice with a graphics queue. Then destroys everything in reverse order. This sequence is present in every Vulkan programme ever written.
💻 Project: D15_VulkanEnv ⏱ ~35 min New: VkInstance · VkPhysicalDevice · VkDevice · VkQueue · validation layers
BUILDS ON: Nothing = Vulkan starting point for every demo ✓
🎯 What You Will See
Terminal: All GPUs listed with name, type, VRAM in MB, and graphics queue family index
Selected GPU: Discrete GPU preferred, fallback to integrated if no discrete present
Validation: VK_LAYER_KHRONOS_validation ENABLED — catches all API mistakes in development
Window: A plain GLFW window opens (proves surface extension works), close it to exit
Shutdown: VkDevice destroyed, then VkInstance — reverse creation order demonstrated
💡 Why This Demo Has No Rendering

The startup sequence is completely independent of rendering. You create the instance and device once at application startup, and they persist until shutdown. Understanding them in isolation — without the complexity of swapchains and pipelines — is the correct way to learn them. Demo 16 adds queues and command pool on top. Demo 17 adds the swapchain.

Vulkan Object Hierarchy — All Four Demos Build This Tree
VkInstance (Demo 15) ← root, connects app to Vulkan runtime
  • VkPhysicalDevice (Demo 15) ← READ ONLY, represents actual GPU hardware
       • VkDevice (Demo 15) ← logical connection, used to create all other objects
            • VkQueue (Demo 16) ← submit command buffers here
            • VkCommandPool + VkCommandBuffer (Demo 16) ← record GPU commands
            • VkSurface + VkSwapchain (Demo 17) ← window + image pool
            •    VkImageView (Demo 17) ← wraps each swapchain image
            •    VkRenderPass + VkFramebuffer (Demo 17) ← render target description
            • VkShaderModule (Demo 18) ← SPIR-V bytecode
            • VkPipeline (Demo 18) ← ALL state baked in at creation
            • VkSemaphore + VkFence (Demo 18) ← frame synchronisation

🔬 Break-to-Learn Experiments
  • Intentionally break the sType field: In VkApplicationInfo, change sType = VK_STRUCTURE_TYPE_APPLICATION_INFO to sType = VK_STRUCTURE_TYPE_DEVICE_CREATE_INFO. Run. Validation layers immediately report: VUID error, wrong sType. Proves validation layers catch even subtle struct misuse.
  • Request a non-existent validation layer: Add "VK_LAYER_LUNARG_fake_layer" to validationLayers. The checkValidationLayerSupport() function returns false and the programme exits cleanly. Shows the two-call enumeration pattern in action.
  • Destroy in wrong order: Move vkDestroyInstance() before any surface or device destroy calls. Validation layers immediately report a lifetime violation. Restore correct order. Shows why destroy-in-reverse-order is mandatory in Vulkan.

💡 VkDevice Is Your Per-Application GPU Handle

In OpenGL, the device was implicit — glDrawArrays() just worked on whatever GPU the driver chose. In Vulkan, VkDevice is an explicit object you create from a chosen VkPhysicalDevice. All subsequent objects (queues, buffers, pipelines, command pools) are created from the device. Queues are not created — they exist inside the physical device. You retrieve them with vkGetDeviceQueue() after the device is created.

🔬 Break-to-Learn Experiments
  • Force a wrong queue family index: In findQueueFamilies(), change the return to always assign index 99 to graphicsFamily. Run. vkCreateDevice() fails with VK_ERROR_INITIALIZATION_FAILED. Validates: Vulkan checks queue family indices strictly against the physical device's actual families.
  • Remove VK_KHR_swapchain from deviceExtensions: Comment out the swapchain extension and leave an empty vector. Device creates fine here — but Demo 17 will crash when it tries to create a swapchain. Shows: extensions must be declared at device creation time, not when first used.
  • Request 2 queues from the same family: Change qi.queueCount = 2. On Intel Iris Xe this may fail (only 1 queue per family available). Check vkGetPhysicalDeviceQueueFamilyProperties — the queueCount field tells you the maximum. Shows: queue count is hardware-limited.

💡 Surface + Swapchain = OpenGL's glfwSwapBuffers() Unpacked

In OpenGL, glfwSwapBuffers() hid: acquiring a free backbuffer, waiting for vsync, flipping front/back. In Vulkan these are three explicit steps: vkAcquireNextImageKHR() gets an image index, your render commands draw into it, vkQueuePresentKHR() flips it. The surface is the OS window abstraction. The swapchain is the pool of images that rotate through front and back.

🔬 Break-to-Learn Experiments
  • Change present mode preference: In choosePresentMode(), move VK_PRESENT_MODE_FIFO_KHR to be the first preference instead of MAILBOX. Rebuild. Frame rate locks to vsync. MAILBOX renders as fast as possible, replacing the queued image. IMMEDIATE has no vsync — tearing may appear on some monitors.
  • Change clear colour: In the render loop, change clearCol = {{{0.04f, 0.08f, 0.18f, 1.0f}}} to orange {{{0.9f, 0.4f, 0.1f, 1.0f}}}. Rebuild. Window changes colour. This is the Vulkan equivalent of glClearColor() — but the value is set per render pass begin, not as global state.
  • Omit the subpass dependency: Remove the VkSubpassDependency from VkRenderPassCreateInfo. Validation layers report: missing synchronisation between acquiring the swapchain image and writing to it. This is the error the dependency prevents.
Demo 18 · Module 12First Triangle + Full Synchronisation
Vulkan First Triangle — Complete Pipeline, SPIR-V Shaders, Full Frame Sync
The Day 4 capstone. A white triangle drawn with the complete Vulkan rendering pipeline: instance, validation layers, physical device selection, logical device, surface, swapchain, image views, render pass, SPIR-V vertex and fragment shaders embedded as bytecode arrays, full VkGraphicsPipelineCreateInfo, framebuffers, command pool, one command buffer per swapchain image, two semaphores (imageAvailable + renderFinished), and one fence. No external .spv files needed — SPIR-V is embedded in the source. Validation layers report zero errors.

Per-Frame Synchronisation — What Happens Every Frame
CPU                              GPU
 |
 |-- vkWaitForFences(inFlight) -->  (block until previous frame done)
 |<----------------------- fence signalled
 |
 |-- vkAcquireNextImageKHR -------->  (give me next swapchain image)
 |<-- imageAvailable semaphore ------
 |
 |-- record: vkCmdBeginRenderPass
 |           vkCmdBindPipeline
 |           vkCmdDraw(3, 1, 0, 0)
 |           vkCmdEndRenderPass
 |
 |-- vkQueueSubmit ---------------->  wait: imageAvailable
 |   (resets + signals inFlight)      execute draw commands
 |                                    signal: renderFinished
 |
 |-- vkQueuePresentKHR ----------->   wait: renderFinished
                                       flip to screen

🔬 Break-to-Learn Experiments
  • Remove the fence wait: Comment out vkWaitForFences() at the start of the frame loop. Run. Validation layers immediately report: “Command buffer is already in use by a previous submission.” Uncomment and the error disappears. This is the exact race condition the fence prevents.
  • Change to wireframe mode: In VkPipelineRasterizationStateCreateInfo, change polygonMode = VK_POLYGON_MODE_FILL to VK_POLYGON_MODE_LINE. Also enable deviceFeatures.fillModeNonSolid = VK_TRUE before device creation. Rebuild the full pipeline. The triangle renders as an outline. In OpenGL this was one glPolygonMode() call — in Vulkan it requires a full pipeline rebuild.
  • Count the lines: Use your editor to count lines in main.cpp. Compare to Demo 1 OpenGL triangle (~80 lines). Every extra line in Demo 18 is one decision OpenGL was making silently. After today, none of those decisions are hidden from you.
